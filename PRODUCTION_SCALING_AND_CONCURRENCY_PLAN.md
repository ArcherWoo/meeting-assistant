# 生产部署与并发稳定性升级总方案

## 1. 文档目的

这份文档用于指导当前项目从“本地可用、单机可跑”升级到“可部署到服务器、支持多人同时在线使用、稳定承载聊天与检索任务”的完整改造。

目标不是一次性做成无限扩展的平台，而是先把以下几个核心问题系统性解决：

- 多人同时访问时，网页和后端服务不容易崩溃
- 每个用户、每个会话彼此隔离，不互相串话或互相污染状态
- 大模型请求可以稳定并发，不因连接管理混乱而拖慢整体体验
- 聊天、知识库、Know-how、附件解析这些耗资源链路在高并发下仍可控
- 后续迭代有明确路线，不会做着做着丢失主线


## 2. 当前背景与核心假设

当前产品形态：

- 前端为 `React + Vite`
- 后端为 `FastAPI`
- 聊天接口为流式 SSE
- 数据存储当前主要基于 `SQLite`
- 会话、消息、Know-how、权限、知识库等能力都已接入后端
- 会有很多用户通过浏览器访问同一个服务器地址
- 同一个用户会开多个新对话
- 多个用户会同时请求大模型输出

本方案默认以下使用场景成立：

- 部署环境是单台或少量服务器，先以单区域部署为主
- 前期允许继续使用兼容 OpenAI 协议的大模型网关
- 聊天是主链路，知识库、Know-how、附件解析是重要增强链路
- 需要优先提升系统稳定性和资源治理，而不是盲目追求功能继续堆叠


## 3. 当前架构现状总结

基于现有代码，当前系统已经具备一定基础，但还没有完成面向“多人在线生产环境”的并发治理闭环。

### 3.1 已有基础

- 聊天主链路已经是异步流式模式，方向正确
- 会话和消息已经入库，不再只存在前端
- 会话访问已经按 `owner_id` 做隔离
- Know-how 和知识库已经开始按当前用户权限过滤
- 大模型请求已经从“每次新建 HTTP 客户端”优化为可复用连接池
- 文件解析中的部分同步库已经通过 `asyncio.to_thread()` 移出事件循环

### 3.2 当前不足

- 大模型调用仍没有统一的全局并发治理中心
- 部分内部服务仍可能自己初始化 `LLMService`，连接与配额未完全统一
- 同一会话还缺少生成锁，可能出现并发写入和并发生成冲突
- `SQLite` 在多人持续写入场景下会成为明显瓶颈
- 文件解析缺少统一线程池治理和任务限流
- 服务端还缺少完整的生产部署参数、健康检查、容量监控和过载保护
- 缺少面向高并发的压测基线和容量评估口径


## 4. 先澄清三个关键概念

### 4.1 “每个新建对话是不是独立线程”

不是。

正确模型是：

- 每个 HTTP 请求是一个异步任务
- 每个会话是一个独立的数据与状态单元
- 同一时刻可以存在很多个会话的异步任务并发执行
- 是否互相影响，不取决于有没有独立线程，而取决于是否有良好的状态隔离、锁和资源治理

系统真正需要的是：

- 会话级状态隔离
- 会话级并发保护
- 请求级取消和超时治理

### 4.2 “每个对话连大模型是不是要多线程”

也不是。

更合适的方式是：

- 共享 HTTP 连接池
- 使用异步 I/O 并发访问大模型
- 用信号量、队列和限流保护上游服务
- 将轻量 LLM 请求和主回答请求分层治理

线程可以用于 CPU 或阻塞型库，例如 PDF、PPT、Excel 解析，但不应该成为 LLM 网络请求的主治理手段。

### 4.3 “网页承载能力强”具体是什么意思

这里至少包含五层：

- 前端静态资源加载稳定
- 后端接口在多人同时请求下不崩溃
- 流式 SSE 不被代理层或应用层错误缓冲
- 大模型、检索、附件解析不会互相抢死资源
- 数据库和存储层不会因为写竞争或锁竞争把整站拖死


## 5. 总体目标架构

目标状态可以概括为一句话：

以“异步并发 + 分层限流 + 会话隔离 + 资源配额 + 生产部署治理”为核心，把聊天系统升级成可稳定服务多人在线使用的应用。

### 5.1 目标原则

- 用户之间数据完全隔离
- 会话之间状态相互独立
- 同一会话内部的写操作可控且有序
- 所有大模型调用统一走一个受控出口
- 重任务和轻任务分层治理
- 服务有明确的过载行为，不把机器拖死
- 每条关键链路都有耗时、错误和并发数可观测


## 6. 需要改造的主链路与问题清单

### 6.1 Web 服务承载能力

当前风险：

- 开发态运行方式不适合生产
- 缺少明确的 worker、超时、连接数、代理配置
- 没有完整的就绪检查和过载保护

需要增强：

- 前后端分离部署
- 静态资源由 Nginx 或等价静态服务承载
- FastAPI 使用生产模式运行
- 通过反向代理处理 TLS、压缩、连接复用和静态文件
- 为 SSE 明确关闭代理缓冲
- 提供健康检查和就绪检查
- 限制请求体大小和长连接行为
- 增加全局过载保护

### 6.2 Chat 并发执行链路

当前风险：

- 同一会话可能被同时发起多个生成请求
- 生成过程中的写库、标题生成、上下文拼装缺乏会话级锁
- 请求取消、重试、重复点击等行为缺少严格治理

需要增强：

- 为每个 `conversation_id` 增加生成锁
- 明确同一会话的并发策略
- 支持取消正在生成的回答
- 防止同一会话的重复落库和消息顺序错乱
- 建立会话级执行状态

### 6.3 LLM 并发治理

当前风险：

- 内部服务可能各自持有自己的 LLM 访问实例
- 缺少统一并发上限
- 缺少用户级限流
- 缺少轻重请求隔离

需要增强：

- 建立统一 LLM Gateway
- 所有模块统一走共享连接池
- 增加全局 Semaphore
- 增加每用户并发上限
- 将请求划分为两类
  - 主回答流式输出
  - 轻量 LLM 任务，例如 gate、judge、rerank、标题生成、测试连通性
- 为两类请求设置不同配额
- 增加失败重试与上游错误治理

### 6.4 检索链路并发治理

当前风险：

- `knowledge / knowhow / skill` 会并发执行，但缺少统一资源预算
- 轻量 LLM 判定与主回答可能争抢同一上游配额
- retrieval 里的 LLM 使用未必完全复用共享网关

需要增强：

- retrieval 链路统一走共享 LLM Gateway
- 为 retrieval 阶段的 LLM 调用使用独立配额
- 对检索阶段设置超时和降级策略
- 当系统繁忙时，优先保主回答，允许关闭部分轻量增强步骤

### 6.5 文件解析与附件处理

当前风险：

- PDF、PPT、Excel 解析属于 CPU/阻塞型任务
- `to_thread()` 虽然避免阻塞事件循环，但没有统一线程池配额
- 高并发上传附件时会争抢 CPU 和内存

需要增强：

- 建立文件解析任务治理层
- 限制同时解析的文件数
- 区分聊天快路径与知识导入重路径
- 针对不同文件类型做快速提取模式
- 对重复文件内容做 hash 缓存
- 对超大文件设定采样和裁剪上限

### 6.6 数据层与持久化

当前风险：

- 当前 `SQLite` 更适合中小规模
- 多人在线持续写消息时会产生写锁竞争
- 随着会话量、消息量增长，查询与维护成本会上升

需要增强：

- 短期继续优化 SQLite 使用方式
- 中期逐步迁移到 `PostgreSQL`
- 将核心高频写入表优先迁出
- 明确索引策略、归档策略、清理策略

### 6.7 权限与隔离

当前风险：

- 聊天、Know-how、知识库、会话都已经有权限逻辑，但高并发下仍需确认无跨用户缓存污染
- 后续引入缓存后，若设计不当，容易把跨用户数据混在一起

需要增强：

- 所有缓存都必须带用户范围或资源范围
- 会话级状态严格绑定 `conversation_id`
- 用户可见资源清单不能做无范围全局缓存
- Chat 自动增强必须继续遵守当前用户权限边界

### 6.8 监控、压测、容量规划

当前风险：

- 现在可以看到单次耗时，但还不够系统化
- 缺少真正的并发压测数据
- 缺少容量上限判断依据

需要增强：

- 完整的结构化日志
- 关键指标打点
- 并发压测脚本
- 每一阶段的容量基线
- 明确告警阈值


## 7. 分阶段升级方案

## Phase 0：方案冻结与基线确认

目标：

- 不急着堆功能，先明确系统边界和目标承载方式

要做的事情：

- 确认生产部署模式
- 确认单机还是多机
- 确认前端静态资源由谁承载
- 确认模型供应商和配额策略
- 确认数据库迁移目标
- 明确高峰场景假设

建议产出：

- 目标部署拓扑图
- 当前链路图
- 配置清单
- 关键瓶颈列表

验收标准：

- 后续每一项改造都有归属和边界


## Phase 1：并发治理最小闭环

目标：

- 先让系统在多人同时使用时“不乱、不抢、不轻易挂”

### 1.1 统一 LLM Gateway

要做的事情：

- 建立全局唯一的 LLM 访问层
- `chat`、`retrieval_planner`、`knowhow_router`、`hybrid_search`、标题生成、连通性测试全部接入同一出口
- 去掉内部服务默认 `LLMService()` 自建实例的行为

验收标准：

- 系统内所有大模型请求都能被统一计数、统一限流、统一观测

### 1.2 增加并发配额

要做的事情：

- 设置全局 LLM 并发上限
- 设置每用户 LLM 并发上限
- 设置轻量请求并发池
- 设置主回答请求并发池
- 请求超出上限时快速失败或排队

验收标准：

- 高并发下不会出现无限堆积把应用拖死

### 1.3 增加会话生成锁

要做的事情：

- 为每个 `conversation_id` 建立生成锁
- 同一会话同时只允许一个主回答生成任务
- 定义冲突策略
  - 拒绝第二个请求
  - 或先取消旧请求，再启动新请求

建议优先选择：

- 默认拒绝第二个请求，并返回清晰提示

验收标准：

- 同一会话不会出现两条回答同时流式输出和写库

### 1.4 增加文件解析限流

要做的事情：

- 为附件提取建立独立 semaphore
- 限制同时解析的文件数
- 为聊天附件和知识导入设置不同优先级

验收标准：

- 多人同时上传附件时，系统仍能响应而不是整体卡住


## Phase 2：生产部署加固

目标：

- 让部署形态真正适合服务器长期运行

### 2.1 部署形态标准化

要做的事情：

- 前端构建为静态资源
- 后端通过生产模式运行
- 使用反向代理承载对外入口
- 为 SSE 正确关闭缓冲

验收标准：

- 流式输出在服务器环境下仍实时可见

### 2.2 健康检查与过载保护

要做的事情：

- 增加 `/health/live`
- 增加 `/health/ready`
- 就绪检查覆盖数据库、LLM 网关状态、关键配置
- 增加系统繁忙保护

验收标准：

- 外部系统可正确判断实例是否可接流量

### 2.3 配置项外置化

要做的事情：

- 将并发数、超时、队列长度、上传限制、解析配额等做成配置
- 区分开发配置和生产配置

验收标准：

- 无需改代码即可调优生产行为


## Phase 3：观测、日志与压测

目标：

- 看得见系统怎么慢、慢在哪里、什么时候会崩

### 3.1 结构化日志

要做的事情：

- 所有 chat 请求打 `request_id`
- 记录 `user_id`、`conversation_id`、`role_id`
- 记录请求开始、结束、取消、超时、上游失败

### 3.2 指标打点

要做的事情：

- 当前在线请求数
- 当前流式回答数
- 当前轻量 LLM 请求数
- 当前附件解析任务数
- LLM 首字耗时
- LLM 总耗时
- retrieval 耗时
- 附件提取耗时
- 写库耗时
- 错误率、超时率、取消率

### 3.3 压测

要做的事情：

- 做普通聊天并发压测
- 做带检索聊天压测
- 做带附件聊天压测
- 做混合场景压测

建议测试梯度：

- 10 并发
- 20 并发
- 50 并发
- 100 并发

验收标准：

- 能明确知道当前单机能承受的并发范围


## Phase 4：数据库升级与数据治理

目标：

- 把最大的中期瓶颈从根上处理掉

### 4.1 短期优化 SQLite

要做的事情：

- 继续使用 `WAL`
- 审查高频查询索引
- 控制单次事务长度
- 评估消息表增长速度

### 4.2 迁移到 PostgreSQL

建议优先迁移表：

- conversations
- messages
- knowhow_rules
- access_grants
- agent_runs

要做的事情：

- 抽象存储接口
- 编写迁移脚本
- 验证兼容性
- 规划回滚策略

验收标准：

- 生产环境写入能力明显优于 SQLite


## Phase 5：缓存与性能增强

目标：

- 在不破坏权限边界的前提下，进一步减少重复开销

要做的事情：

- 对模型列表做短缓存
- 对附件文本提取结果做 hash 缓存
- 对知识检索候选做短期缓存
- 对 Know-how 分类画像或规则摘要做范围化缓存

注意事项：

- 所有缓存必须带作用域
- 严禁将不同用户的可见资源混用


## 8. 各模块详细改造项

## 8.1 Chat 模块

需要改造：

- 增加会话级生成锁
- 明确生成中的状态管理
- 支持取消、超时、重复发起保护
- 在响应元数据中继续保留阶段耗时
- 将重任务和轻任务的上下文增强策略分级

需要特别检查：

- 同会话连续点击发送
- 用户刷新页面后重新连接
- SSE 断开后的状态清理


## 8.2 LLM 模块

需要改造：

- 统一入口
- 并发池
- 用户级配额
- 上游错误分类
- 非流式轻请求重试
- 连接池监控

需要特别检查：

- 大量并发流式请求下连接池是否耗尽
- `/models` 或探测接口是否抢占主回答资源
- 测试连通性是否可以走轻量专用通道


## 8.3 Retrieval 模块

需要改造：

- planner、knowhow、knowledge、skill 的 LLM 调用全部纳管
- 设置 retrieval 预算
- 系统繁忙时允许降级关闭部分增强能力

需要特别检查：

- retrieval 阶段的 LLM 是否和主回答争同一个并发池
- Know-how 与知识库的缓存是否带权限作用域


## 8.4 文件解析模块

需要改造：

- 建立解析线程池策略
- 统一快路径和重路径
- 对 PDF、PPT、Excel、DOCX 制定文件级预算
- 对超大文件做裁剪

需要特别检查：

- 多人同时上传多个文件的峰值情况
- 文件 hash 缓存命中后是否可直接复用


## 8.5 会话与消息模块

需要改造：

- 会话级锁
- 消息顺序一致性
- 并发写保护
- 会话状态可恢复

需要特别检查：

- 一个用户打开多个标签页同时操作同一会话
- 流式生成中删除会话或切换会话


## 8.6 Know-how 与知识库模块

需要改造：

- 在高并发场景下继续严格遵守权限边界
- 避免引入无范围缓存
- 让 Chat 增强在系统繁忙时可降级

需要特别检查：

- 用户 A 的缓存不会命中到用户 B 的可见范围
- 组内共享和个人私有规则的检索不会串数据


## 9. 配置项规划

建议新增或整理以下配置：

- `LLM_GLOBAL_CONCURRENCY`
- `LLM_STREAM_CONCURRENCY`
- `LLM_LIGHTWEIGHT_CONCURRENCY`
- `LLM_PER_USER_CONCURRENCY`
- `CHAT_CONVERSATION_LOCK_TIMEOUT_MS`
- `ATTACHMENT_PARSE_CONCURRENCY`
- `ATTACHMENT_FAST_MODE_MAX_BYTES`
- `ATTACHMENT_EXCEL_MAX_ROWS`
- `ATTACHMENT_EXCEL_MAX_COLS`
- `RETRIEVAL_TIMEOUT_MS`
- `REQUEST_MAX_BODY_BYTES`
- `SSE_IDLE_TIMEOUT_MS`
- `HEALTHCHECK_DEGRADED_THRESHOLD`

要求：

- 均可通过环境变量或配置文件调整
- 需要有开发默认值和生产建议值


## 10. 验收指标与目标口径

### 10.1 稳定性指标

- 服务长时间运行不因并发请求轻易崩溃
- 高并发下不会出现明显串话、乱序、重复写库
- 系统繁忙时能优雅失败，而不是全站拖死

### 10.2 性能指标

- 普通短问首字时间稳定
- 轻量测试连通性可快速返回
- 带附件聊天在合理文件大小下有可接受前置等待
- retrieval 阶段在繁忙场景下可降级

### 10.3 隔离指标

- 不同用户不可见对方会话
- 不同用户不可命中超范围 Know-how
- 同一用户多个会话互不干扰
- 同一会话并发请求有明确处理结果


## 11. 风险与注意事项

### 11.1 不要把“并发”误做成“多线程 everywhere”

错误方向：

- 每个对话开线程
- 每次 LLM 请求开线程
- 每个附件开不受控线程

正确方向：

- 网络请求走异步并发
- 阻塞文件解析走受控线程池
- 通过锁、Semaphore、队列治理资源

### 11.2 不要过早引入复杂分布式架构

当前更合适的路径：

- 先把单机单实例治理做好
- 再做多 worker
- 再根据数据决定是否做多实例和数据库迁移

### 11.3 不要牺牲权限边界换性能

所有缓存、加速、摘要、检索候选都必须继续尊重用户权限、组权限和管理员权限。


## 12. 推荐实施顺序

最推荐的实际推进顺序如下：

1. 建立统一 LLM Gateway
2. 增加全局并发池、轻重请求分池、每用户并发上限
3. 给 chat 增加会话级生成锁
4. 给文件解析增加并发上限和线程池治理
5. 外置并发、超时、上传限制等配置
6. 补健康检查、结构化日志和核心指标
7. 做第一轮压测
8. 根据压测结果决定是否推进 PostgreSQL 迁移
9. 再做缓存和高级性能增强

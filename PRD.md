## 一、产品概述

### 1.1 产品定位
Meeting Assistant 是一款面向**非技术用户**的 AI 桌面客户端。核心能力是通过自然语言对话处理 PPT 文件、总结会议内容，并将重复性工作流封装为可复用的 **Skill（技能）**。

### 1.2 核心理念
- **自然语言驱动**：用户无需拖拽、无需写代码，通过对话即可创建和使用技能
- **Markdown-as-Code**：借鉴 Claude Code / OpenClaw 的 `SKILL.md` 模式，技能以人类可读的 Markdown 文件定义
- **三模式架构**：Copilot（问答）→ Skill Builder（生成技能）→ Agent Executor（执行任务）

### 1.3 目标用户
| 用户类型 | 典型场景 |
|---------|---------|
| **⭐ 采购人员** | **上传采购 PPT → 会前材料预审 → 历史价格对比 → 老板关注点检查** |
| 行政/秘书 | 上传会议 PPT → 自动生成会议纪要 |
| 项目经理 | 批量处理周报 PPT → 提取关键进展 |
| 培训讲师 | PPT 内容提取 → 生成培训总结/考核题 |
| 销售团队 | 客户 PPT 分析 → 提取关键信息 |

> **首要场景**：采购会前材料预审是本产品的核心落地场景，所有架构设计优先保障此场景的完整体验。

### 1.4 技术栈
| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | **React + TypeScript + Vite** | 现代组件化 UI，Vite 构建 |
| 后端 | **Python FastAPI** | 本地 HTTP 服务，承载所有 AI/数据/文件处理逻辑 |
| PPT 解析 | **Docling (IBM)** + **python-pptx** | Docling 主引擎（AI 布局分析 + 高精度表格提取 → Markdown/JSON），python-pptx 辅助提取元数据与备注 |
| LLM 接入 | **OpenAI 兼容协议** | 流式 SSE，URL + Bearer Token |
| Embedding | **BGE-M3** (ONNX Runtime) | 本地 Embedding 模型，768 维，中英文优秀，CPU 即可运行 |
| 向量存储 | **LanceDB** | 嵌入式向量数据库，本地零配置 |
| 结构化存储 | **SQLite** | 轻量关系型数据库，结构化数据查询 |
| 样式 | **TailwindCSS** | 现代简洁高级感 |
| 状态管理 | **Zustand** | 轻量级状态管理 |
| 安全 | **Fernet / keyring** | API Key 加密存储，本地数据加密 |

> **架构总览**：前端通过 Vite 开发服务器运行，通过 `http://localhost:8765` 与 Python FastAPI 后端通信。所有 AI 推理、文件解析、知识库检索、Skill 执行均在 Python 端完成，前端仅负责 UI 渲染与交互。

### 2.2 Copilot Mode（💬 连续问答）
- **用途**：通用对话、知识问答、文档理解
- **System Prompt**：加载 `AGENT.md`（人格） + `USER.md`（用户画像） + `MEMORY.md`（长期记忆）
- **特点**：
  - 多轮上下文保持
  - 支持附件上传（PPT/PDF/图片/文本）
  - 不调用工具，纯对话模式
  - 支持提示词增强

### 2.3 Skill Builder Mode（🔧 技能生成 + AI Coding）
- **用途**：通过澄清式对话，将用户需求转化为 `.skill.md` 文件 **及配套的可执行 Tool 代码**
- **System Prompt**：加载 `ARCHITECT.md`（架构师人格）+ `CODEGEN.md`（代码生成规范）
- **核心能力**：
  - 生成 `.skill.md` 技能描述文件
  - **生成 `.tool.ts` / `.tool.py` 可执行工具代码**（AI Coding，详见第十一章）
  - **生成 Function Call JSON Schema**（符合 OpenAI 规范）
  - 自动注册新 Tool 到动态工具系统
- **工作流程**：
  ```
  用户描述需求 → Agent 澄清细节（3-5 轮对话）
       → 生成 SKILL.md 草稿
       → 生成配套 Tool 代码（.tool.ts/.tool.py）+ Function Call Schema
       → 沙箱验证代码安全性
       → 用户确认/修改
       → 保存到 skills/ + tools/ 目录
       → 热加载注册 + 自动索引更新
  ```
- **SKILL.md 文件结构**：
  ```markdown
  # Skill: PPT 会议纪要生成器

  ## 描述
  从上传的 PPT 文件中提取内容，生成结构化会议纪要。

  ## 触发条件
  - 关键词: "会议纪要", "总结PPT", "会议总结"
  - 输入类型: .pptx 文件

  ## 输入参数
  - file: PPT 文件路径（必需）
  - format: 输出格式 [markdown | text]（默认: markdown）
  - language: 输出语言（默认: 中文）

  ## 执行步骤
  1. 调用 `pptx_parser` 工具提取 PPT 全部内容
  2. 按幻灯片顺序整理文本、图表描述
  3. 使用 LLM 生成结构化摘要：
     - 会议主题
     - 关键议题（按幻灯片章节）
     - 决策事项
     - 待办事项（Action Items）
  4. 输出最终会议纪要

  ## 依赖工具
  - pptx_parser
  - llm_summarizer
  ```

### 2.4 Agent Executor Mode（🤖 任务执行）
- **用途**：自动匹配用户意图到已有 Skill，调用工具链执行
- **System Prompt**：加载 `AGENT.md` + 可用 Skills 索引 + 可用 Tools 列表
- **执行流程**：
  ```
  用户输入 → 意图识别 → 匹配 Skill
       → 参数提取/确认
       → 逐步执行（实时进度反馈）
       → 返回结果
  ```
- **内置工具（Tools）**：
  | 工具名 | 功能 |
  |--------|------|
  | `pptx_parser` | 解析 PPT 文件，提取文本/图片/表格 |
  | `procurement_extractor` | 从 PPT 中提取结构化采购字段（供应商/品类/价格等） |
  | `knowledge_query` | 混合检索知识库（SQLite 精确查询 + LanceDB 语义检索） |
  | `file_reader` | 读取本地文件内容 |
  | `file_writer` | 写入文件到本地 |
  | `llm_summarizer` | 调用 LLM 进行内容总结 |
  | `web_search` | 联网搜索补充信息 |
  | `clipboard` | 复制内容到剪贴板 |

### 2.5 模式切换机制
| 切换方式 | 说明 |
|---------|------|
| **UI 切换栏** | 顶部 Tab 或 Segmented Control 手动切换 |
| **斜杠命令** | `/copilot` `/builder` `/agent` |
| **自动检测** | Agent 根据用户意图自动建议切换 |

**上下文策略**：
- Copilot → Builder：保留上下文（可能正在讨论需求）
- Builder → Agent：传递新建 Skill 信息，清空对话
- Agent → Copilot：传递执行结果摘要
- 手动切换：提示用户确认

### 3.2 设计语言

#### 配色方案
| 元素 | Light Mode | Dark Mode |
|------|-----------|-----------|
| 背景 | `#FAFAFA` | `#1A1A2E` |
| 侧边栏 | `#F5F5F5` | `#16213E` |
| 卡片/气泡 | `#FFFFFF` | `#0F3460` |
| 主色调 | `#6C63FF`（紫蓝） | `#6C63FF` |

| 文字次色 | `#8E8E93` | `#A0A0A0` |
| 分割线 | `#E5E5EA` | `#2A2A4A` |

#### 设计原则
- **毛玻璃效果（Glassmorphism）**：侧边栏和弹窗使用 `backdrop-filter: blur(20px)` + 半透明背景
- **圆角统一**：卡片 `12px`，按钮 `8px`，输入框 `10px`
- **阴影层次**：`0 2px 8px rgba(0,0,0,0.06)` 轻阴影，`0 8px 32px rgba(0,0,0,0.12)` 浮层阴影
- **动效**：所有过渡 `200ms ease-out`，列表项 stagger 动画
- **间距系统**：基于 `4px` 网格（4/8/12/16/24/32/48）

### 3.4 关键交互设计

#### 3.4.1 附件上传
- **触发方式**：点击 📎 按钮 / 拖拽文件到输入区 / Ctrl+V 粘贴
- **支持格式**：`.pptx` `.ppt` `.pdf` `.docx` `.txt` `.md` `.csv` `.xlsx` `.png` `.jpg`
- **上传反馈**：文件缩略图 + 文件名 + 大小 + 删除按钮
- **PPT 特殊处理**：上传后自动调用后端 Docling + python-pptx 双引擎预解析，显示幻灯片数量与表格数量

#### 3.4.2 提示词增强（Prompt Enhancement）
- **触发方式**：点击 ✨ 按钮 或 快捷键 `Ctrl+E`
- **工作流程**：
  ```
  用户输入原始提示词 → 点击增强 → 调用 LLM 优化提示词
       → 显示优化后的版本（diff 高亮）
       → 用户确认使用 / 编辑 / 取消
  ```
- **增强策略**：
  - 补充上下文和约束条件
  - 明确输出格式要求
  - 添加 few-shot 示例（如适用）

#### 3.4.3 模式切换
- **视觉反馈**：切换时输入区域边框颜色变化（💬蓝 / 🔧橙 / 🤖绿）
- **引导提示**：首次切换到新模式时显示简短说明 tooltip
- **快捷键**：`Ctrl+1/2/3` 快速切换

## 5½、核心场景：采购会前材料预审

> 本节描述产品的**首要落地场景**——采购人员在开会前，使用 Agent 对采购 PPT 进行自动化预审。

### 5½.1 用户画像

| 属性 | 描述 |
|------|------|
| **角色** | 采购专员 / 采购经理 |
| **技术水平** | 非技术人员，熟练使用 Office，不会写代码 |
| **日常痛点** | 每次开会前需手动翻阅历史采购记录对比价格，耗时且易遗漏老板关注的要点 |
| **核心诉求** | 上传最新采购 PPT → 系统自动对比历史数据 + 检查老板关注点 → 输出预审报告 |
| **数据规模** | 几十到几百份历史采购 PPT，未来可能上千份 |
| **安全要求** | 采购数据高度敏感，必须纯本地处理，不可上传云端 |

### 5½.2 预审工作流

```
用户上传采购 PPT
    │
    ▼
┌─ 结构化提取 ──────────────────────────────────────┐
│  python-pptx 解析 → LLM 提取结构化字段：           │
│  · 供应商名称、品类、单价、数量、总价              │
│  · 合同条款、交付周期、付款方式                    │
│  · 技术参数、资质要求                              │
└───────────────────────────┬────────────────────────┘
                            │
    ┌───────────────────────┼───────────────────────┐
    ▼                       ▼                       ▼
┌─ 历史对比 ─┐    ┌─ 老板关注点检查 ─┐    ┌─ 类似案例 ─┐
│ SQLite 查询  │    │ KNOWHOW 规则匹配 │    │ LanceDB    │
│ 同品类均价   │    │ 逐项检查是否覆盖 │    │ 语义检索   │
│ 价格偏差%    │    │ 标记缺失项       │    │ 相似采购   │
└──────┬───────┘    └──────┬───────────┘    └──────┬─────┘

              ┌─ 生成预审报告 ──────────────┐
              │  · 价格合理性评估            │
              │  · 老板关注点覆盖情况（✅/❌）│

              └─────────────────────────────┘
```

### 5½.3 内置 Skill：采购会前预审

```markdown
# Skill: 采购会前材料预审

## 描述
对采购 PPT 进行全面预审，包括价格合理性分析、老板关注点检查、历史案例关联，
输出结构化预审报告，帮助采购人员在会前做好充分准备。

## 触发条件
- 关键词: "采购预审", "会前审查", "采购分析", "预审报告", "审查采购材料"
- 输入类型: .pptx 文件

## 输入参数
- file: 采购 PPT 文件路径（必需）
- check_history: 是否对比历史价格（默认: true）
- check_knowhow: 是否检查老板关注点（默认: true）
- output_format: 输出格式 [markdown | html]（默认: markdown）

## 执行步骤
1. 调用 `pptx_parser` 解析 PPT 全部内容
2. 调用 `procurement_extractor` 提取结构化采购字段：
   - 供应商信息（名称、资质、联系方式）
   - 采购品类与规格参数
   - 报价明细（单价、数量、总价、税率）
   - 合同条款（交付周期、付款方式、违约条款）
   - 技术要求与验收标准
3. 若 check_history=true：
   - 调用 `knowledge_query` 查询同品类历史采购均价
   - 计算本次报价与历史均价的偏差百分比
   - 标记偏差超过 ±15% 的异常项
4. 若 check_knowhow=true：
   - 加载 KNOWHOW 规则库（老板关注点清单）
   - 逐项检查 PPT 内容是否覆盖每个关注点
   - 对未覆盖的关注点标记为 ❌ 并给出补充建议
5. 调用 `knowledge_query` 语义检索历史类似采购案例（Top 3）
6. 综合以上分析，使用 LLM 生成预审报告

## 输出格式
### 📋 采购会前预审报告
**文件**: {filename}
**预审时间**: {datetime}
**预审结论**: ✅ 建议通过 / ⚠️ 需关注 / ❌ 建议驳回

#### 一、基本信息摘要
| 字段 | 内容 |
|------|------|
| 供应商 | xxx |
| 采购品类 | xxx |
| 总金额 | xxx |
| ... | ... |

#### 二、价格合理性分析
| 品类 | 本次单价 | 历史均价 | 偏差 | 判定 |
|------|---------|---------|------|------|
| A    | ¥100    | ¥95     | +5.3% | ✅ 正常 |
| B    | ¥200    | ¥150    | +33.3% | ❌ 偏高 |

#### 三、老板关注点检查
| 关注点 | 是否覆盖 | 说明 |
|--------|---------|------|
| 供应商资质证明 | ✅ | 第3页包含营业执照和资质证书 |
| 历史合作记录 | ❌ | PPT 中未提及，建议补充 |
| 交付时间节点 | ✅ | 第7页明确了里程碑计划 |

1. **2025-Q3 办公设备采购**（相似度 92%）— 最终成交价低于报价 8%
2. **2025-Q1 IT 设备采购**（相似度 85%）— 注意交付延迟风险

- ⚠️ 缺少历史合作记录，建议补充或要求提供客户案例

```

## 5⅔、Know-how 积累系统

> **核心理念**：让系统"越用越好用"。通过用户反馈不断沉淀业务知识（know-how），使 Agent 逐渐理解特定用户/团队的工作习惯和判断标准。

### 5⅔.1 什么是 Know-how

Know-how 是用户在使用过程中积累的**业务经验规则**，例如：
- "老板特别关注供应商是否有 ISO 9001 认证"
- "IT 设备采购超过 50 万必须有三家以上报价对比"
- "交付周期超过 90 天的项目需要特别标注风险"
- "张总不喜欢 PPT 里出现模糊的交付时间描述，比如'尽快'"

这些规则**不是通用知识**，而是特定组织/个人的工作偏好和判断标准。

### 5⅔.2 Know-how 存储结构

Know-how 以 JSON 格式存储在本地 `data/knowhow.json`，每条规则包含：

```json
{
  "knowhow": [
    {
      "id": "kh_001",
      "category": "采购预审",
      "rule": "供应商必须提供 ISO 9001 质量管理体系认证",
      "source": "user_feedback",
      "created_at": "2026-03-15T10:30:00",
      "updated_at": "2026-03-17T14:20:00",
      "confidence": 0.95,
      "hit_count": 12,
      "last_hit": "2026-03-17T14:20:00",
      "tags": ["供应商资质", "ISO认证"],

    },
    {
      "id": "kh_002",
```

### 5⅔.4 Know-how 在 Skill 执行中的应用

当 Agent 执行采购预审 Skill 时：

1. **加载规则**：从 `knowhow.json` 中加载 `category="采购预审"` 的所有规则
2. **逐项检查**：将每条规则与 PPT 内容进行匹配
3. **标记结果**：覆盖的标 ✅，未覆盖的标 ❌ 并给出建议
4. **更新统计**：命中的规则 `hit_count += 1`，`confidence` 微调上升

4. **纯本地运行**：所有 AI 模型本地推理（CPU 即可，GPU 加速可选），完全满足数据隐私要求
5. **MIT 许可证**：商业友好，无使用限制
6. **活跃维护**：IBM 持续投入，AAAI 2025 发表论文，社区活跃（GitHub 15k+ stars）

### 6.3 Docling 集成代码示例
```python
from docling.document_converter import DocumentConverter
from docling.datamodel.base_models import InputFormat
from pptx import Presentation
import json

class PPTParser:
    """双引擎 PPT 解析器：Docling（主）+ python-pptx（辅）"""

    def __init__(self):
        self.converter = DocumentConverter(
            allowed_formats=[InputFormat.PPTX, InputFormat.PDF, InputFormat.DOCX]
        )

    async def parse(self, file_path: str) -> dict:
        # 1. Docling 主解析 → Markdown + 结构化数据
        result = self.converter.convert(file_path)
        doc = result.document
        markdown = doc.export_to_markdown()
        tables = [t.export_to_dataframe() for t in doc.tables]

        # 2. python-pptx 辅助提取元数据和备注
        prs = Presentation(file_path)
        metadata = self._extract_metadata(prs)
        notes = self._extract_notes(prs)

        # 3. 合并输出
        return {
            "metadata": metadata,
            "slides": self._merge_slides(doc, prs, notes),
            "full_markdown": markdown,
            "extraction_stats": self._stats(doc)
        }

    def _extract_metadata(self, prs: Presentation) -> dict:
        props = prs.core_properties
        return {
            "title": props.title or "",
            "author": props.author or "",
            "created": str(props.created) if props.created else "",
            "modified": str(props.modified) if props.modified else "",
            "slide_count": len(prs.slides),
            "parser": "docling+python-pptx"
        }

    def _extract_notes(self, prs: Presentation) -> list[str]:
        return [
            slide.notes_slide.notes_text_frame.text
            if slide.has_notes_slide else ""
            for slide in prs.slides
        ]
```

#### 11.3.3 热加载实现
```typescript
// Tool Registry Manager
class ToolRegistry {
  private tools: Map<string, ToolEntry> = new Map();

  // 注册新工具（无需重启应用）
  async registerTool(toolDir: string): Promise<void> {
    const meta = await readJSON(`${toolDir}/*.meta.json`);
    const schema = await readJSON(`${toolDir}/*.schema.json`);

    // 1. 验证 schema 合法性
    validateFunctionCallSchema(schema);

    // 2. 动态加载执行模块
    const executor = await import(`${toolDir}/${meta.name}.tool.ts`);

    // 3. 注册到内存索引
    this.tools.set(meta.name, { meta, schema, executor });

    // 4. 更新 registry.json
    await this.persistRegistry();

    // 5. 通知 LLM 层更新可用 tools 列表
    this.emit('tools-updated', this.getSchemaList());
  }

  // 获取所有可用的 Function Call Schema（传给 LLM）
  getSchemaList(): FunctionCallSchema[] {
    return Array.from(this.tools.values())
      .filter(t => t.meta.enabled)
      .map(t => t.schema);
  }

  // 执行工具
  async executeTool(name: string, input: any, ctx: ToolContext): Promise<ToolResult> {
    const tool = this.tools.get(name);
    if (!tool) throw new Error(`Tool not found: ${name}`);
    return await this.sandbox.run(tool.executor, input, ctx);
  }
}
```

```python
    extraction_prompt = """
    请从以下 PPT 内容中提取所有采购相关的结构化信息。
    对每个采购项，提取以下字段（JSON 格式）：
    - supplier: 供应商名称
    - category: 采购品类
    - item_name: 具体品名
    - specification: 规格参数
    - unit_price: 单价（数字）
    - quantity: 数量（数字）
    - unit: 单位
    - total_price: 总价（数字）
    - delivery_days: 交付天数（数字）
    - payment_terms: 付款方式

    PPT 内容：
    {ppt_content}
    """
    structured_data = await llm_extract(extraction_prompt, ppt_content)

    # 4. 写入 SQLite
    await sqlite_insert_items(structured_data)

    # 5. 文本分块 + Embedding → 写入 LanceDB
    chunks = chunk_by_slide(ppt_content)
    embeddings = await embed_texts([c.text for c in chunks])
    await lancedb_insert(chunks, embeddings)

    # 6. 记录导入状态
    await db.record_import(file_path, file_hash, len(structured_data))

    return IngestResult(status="success", items_count=len(structured_data))
```

### 12.5 Hybrid Search 查询流程

```python
async def hybrid_query(query: str, filters: dict = None) -> QueryResult:
    """混合检索：SQLite 精确查询 + LanceDB 语义检索"""

    # 1. LLM 意图分析：判断查询类型
    intent = await analyze_query_intent(query)
    # intent.type: "price_compare" | "similar_case" | "supplier_info" | "mixed"

    # 2. 结构化查询（SQLite）
    sql_results = []
    if intent.needs_structured_query:
        sql = intent.generate_sql()  # LLM 生成 SQL
        # 例如: SELECT category, AVG(unit_price) as avg_price
        #       FROM procurement_items WHERE category = '办公设备'
        #       GROUP BY category
        sql_results = await sqlite_execute(sql)

    # 3. 语义检索（LanceDB）
    vector_results = []
    if intent.needs_semantic_search:
        query_embedding = await embed_text(query)
        vector_results = await lancedb_search(
            query_embedding,
            top_k=5,
            filter=filters  # 可选：按品类/供应商过滤
        )

    # 4. 合并结果 + LLM 综合分析
    combined = merge_results(sql_results, vector_results)
    analysis = await llm_analyze(query, combined)

    return QueryResult(
        structured_data=sql_results,
        similar_cases=vector_results,
        analysis=analysis
    )
```

### 12.7 Embedding 模型选择

| 方案 | 模型 | 维度 | 大小 | 优势 | 劣势 |
|------|------|------|------|------|------|
| **本地（默认）** | **BGE-M3** (ONNX Runtime) | 768 | ~600MB | 完全离线、隐私安全、中英文效果优秀 | 首次加载需下载模型（~600MB） |
| 云端（可选） | text-embedding-3-small（OpenAI 兼容） | 1536 | — | 质量略高、无需本地资源 | 需要网络、数据过云端 |

> **默认使用本地 BGE-M3 模型**，所有 Embedding 计算均在本地完成，采购数据不出设备。用户也可在设置中切换为云端 Embedding（复用已配置的 LLM API Key）。
>
> **BGE-M3 选择理由**：
> - 由 BAAI（智源研究院）开源，中文效果业界领先
> - 支持多语言（中/英/日/韩等 100+ 语言）
> - 支持多粒度检索（Dense + Sparse + ColBERT）
> - ONNX Runtime 推理，CPU 即可运行（无需 GPU）
> - 首次启动自动下载模型到 `~/.meeting-assistant/models/bge-m3/`，后续离线可用
>
> **性能参考（CPU, Apple M1/Intel i7）**：
> - 单条文本 Embedding：~50ms
> - 批量 100 条：~2s
> - 模型加载（首次）：~3s

```python
    "backup_count": 5,            # 保留最近 5 个日志文件
    "level": "INFO",              # 默认级别
    "format": "%(asctime)s [%(levelname)s] %(name)s: %(message)s",
}

# 日志分类
# app.log       — 应用主日志（启动/关闭/错误）
# api.log       — LLM API 调用日志（不含内容，仅记录 token 用量和耗时）
# skill.log     — Skill 执行日志
# knowledge.log — 知识库操作日志（导入/查询/索引）
```

**日志脱敏规则**：
- ❌ 不记录：API Key、用户对话内容、PPT 文件内容
- ✅ 记录：API 调用耗时、Token 用量、错误堆栈、Skill 执行状态
